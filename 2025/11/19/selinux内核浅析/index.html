1.基础概念

1.1 domain和type

当前有一条selinux策略：

位置 内容 角色 解释
第 1 个标识符 init domain（进程 type） 作用在主体（subject），即进程
第 2 个标识符 fsck_exec type（对象的标签） 作用在客体（object），即文件
第 3 个标识符 file class（对象类型） 被访问的对象属于 file 类
花括号权限 { read write } permissions 对 file class 的权限

什么是domian域？

SELinux 源自 Flask/Security-Enhanced Linux 的 Type Enforcement 模型。在 Type Enforcement 中：

  • Subject（进程）属于 domain
  • Object（文件等）属于 type

需要注意：一个进程永远只有一个 domain

1.2 domain和scontext的关系

我们经常在日志中看到avc denied，例如：audit(0.0:67): avc: denied { write }for path=”/dev/block/vold/93:96” dev=”tmpfs” ino=1263 scontext=u:r:kernel:s0 tcontext=u:object_r:block_device:s0 tclass=blk_file permissive=0

我们配规则的时候肯定这样配：allow kernel block_device:blk_file { write };

根据上面的知识：

  • kernel是进程domain，是进程的type
  • block_device是type，是对象的标签

1.4 AVC

在 SELinux 内核中，state->avc 是 访问向量缓存（Access Vector Cache, AVC） 的核心数据结构。

1️⃣ 概念

  • AVC（Access Vector Cache）：
    • 一个缓存，用来存储 主体 SID 与客体 SID 在某个 class 上的权限决策。
    • 目的是 避免每次访问都遍历 policydb 计算权限，提高性能。
  • state->avc 是 selinux_state 结构体中的一个指针，指向 AVC 实例：
  • 凡是需要判断权限的地方，都会通过 avc_has_perm() / avc_has_perm_noaudit() 访问 state->avc。

2️⃣ AVC 缓存里存什么

每个缓存节点（struct avc_node）对应一个 源 SID / 目标 SID / 对象 class 三元组：

  • ae.avd 里面存：
    • allowed 位图 → 已允许的权限
    • auditallow / auditdeny → 审计控制
  • 可能还有 扩展权限（xperms）
  • AVC 是 hash table，按 (ssid, tsid, tclass) 查找

3️⃣ AVC 的工作流程

  1. 权限检查时：
  • 如果缓存命中 → 直接返回 avd
  • 如果未命中 → 调用 avc_compute_av() 计算权限，并插入 AVC
  1. 计算权限：
  • 调用 context_struct_compute_av()
  • 根据 policydb（TE rules、constraints、role/type transition）
  • 得到 allowed/audit 权限
  1. 缓存结果：
  • 下次同样的 SID pair / class，直接从 state->avc 查
  • 避免重复计算

4️⃣ 为什么 AVC 很重要

  • SELinux 内核每天要做大量权限检查（文件读写、进程 exec、socket 操作）
  • policydb 遍历 + ebitmap 展开比较耗 CPU
  • AVC 缓存三元组决策 → 几乎是 O(1) 查找
  • 提高系统性能，尤其在 Android 上，频繁访问文件和进程间通信的场景

Q1：policydb是什么？

在 SELinux 内核中，policy->policydb 是 SELinux 策略的核心数据库，保存了全部 TE（Type Enforcement）、属性、角色、类型、类别、约束等信息，用于在内核中进行权限判断。

Q2：sid和scontext的关系是什么？

SID 是一个整数 ID，用来索引； scontext（struct context）是真正的安全上下文结构（user:role:type）。

因为 context（user/role/type/range 字符串）很复杂，不能直接存储在 inode、task_struct、packet 里，代价太大。

所以 SELinux 用 SID 做成一个轻量句柄：“context 作为值存一份，SID 是 key，用 key 引用它。”

这样带来几个好处：

  • 更快（比较 SID 比比较字符串快）
  • 节省内存（context 结构共享）
  • 政策变化时只更新 sidtab，不必修改所有对象

这里很有意思了，还记得在1.3章节里面说domain可以属于多个attribute，type也可以属于多个attribute，这里真正的allow规则就是这么索引的，不是简单的只看domain对type的访问权限
