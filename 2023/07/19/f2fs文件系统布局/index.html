2.整体布局

这张图来自于论文：《F2FS: A New File System for Flash Storage》

可以看到整体布局是常规的文件系统布局方式：超级块 + 元数据区 + 用户数据区；

  • 元数据区分为了CP、SIT、NAT、SSA四个区域，这四个区域都是以Segment为单位进行对齐的。
  • 因f2fs为日志结构文件系统LFS，所以其用户数据区也称为日志区Main Area。f2fs不同于传统的LFS将所有的数据都存储在一个日志里面，f2fs将其划分为了Node Segment和Data Segment，又根据其是否经常改动划分为了Hot、Warm和Cold，因此总共是6种形式。

f2fs基本数据单位

  • block: F2FS的数据存储的基本单位是block，大小为4KB，整个flash设备被格式化为多个block组成的结构。很多数据结构也被设计为4KB的大小，这是因为很多flash设备单次IO的读写都是基于4KB的倍数进行。
  • segment: segment是管理block的结构，一个segment的大小是512个block，也就是2MB。
  • section: 默认情况下一个segment等于一个section，section是GC的基本操作单位，每次GC都会从section中选出特定的segment进行回收。F2FS将section分为了6类，分别是hot-node，warm-node，cold-node，hot-data，warm-data，cold-data，hot->cold表示了数据的从高到低的修改频率，通过不同类型section：进行gc的时候可针对使用hot的section进行gc，以降低gc的时间开销。
  • zone: 默认情况一个zone等于一个section，与物理设备有关，大部分情况下用不上

值得注意的是，f2fs的superblock不是从0x0开始，而是预留了1KB，即其偏移量为0x400

  • magic：f2fs的魔数字，用来检查SuperBlock是否破损，其内容固定为0xF2F52010。

  • major_ver、minor_ver：这两个是Linux用于标识一个块设备的主次号，主次号可以确定系统中的每一个块设备。

  • log_sectorsize：块设备中一个扇区大小，普通设备是512，此处是对2取对数后的结果，所以是9。

  • log_sectors_per_block：一个block对应扇区数，并对2取对数后的结果，一个block是4KB，而sector是512，所以结果是3。

  • log_blocksize：一个block大小对2取对数的结果，也即log2(4096)，结果是12。

  • log_blocks_per_seg：一个segment对应的block数目，并对2取对数的结果，结果是9。

  • segs_per_sec、secs_per_zone：分别是一个section对应的segmeng数量和一个zone对应的section数量，默认两者均为1。

  • crc_offset、crc：SuperBlock中的crc字段偏移量，crc则是该SuperBlock结构体的crc计算结果，用于检查SuperBlock的正确性。其中这两个字段由F2FS_FEATURE_SB_CHKSUM标记位控制，如果设置了该标记，mkfs才会真正写入这两个字段的内容。

  • block_count：整个块设备的block个数，需要做4KB对齐，如果对应多个设备，计算的是所有设备的block个数。

  • section_count：Main Area对应的section个数，如果是多设备，需要全部计算。

  • segment_count_ckpt：CP区域所占的segment数，CP区域固定占2个segment。

  • segment_count_sit：SIT区域所占用的segment数，这个是根据segment_count的值确定的，一个f2fs_sit_entry结构体占74字节，所以一个block可容纳56个f2fs_sit_entry，也即一个segment可容纳28672个f2fs_sit_entry，对应的是56G。但由于SIT有一个备份区域，所以每56G的大小就需要2个segment。注意：SIT区域管理系统中所有的segment，包括CP区域和SIT区域自身。

  • segment_count_nat：NAT区域所占用的segment数，对应的是从NAT区域开始的所有block（包括NAT、SSA、Node、Data区域）。一个f2fs_nat_entry大小是9字节，所以一个block对应455个f2fs_nat_entry。同样，NAT区域也有一个备份区。

  • segment_count_ssa：SSA区域所占用的segmeng数，一个SSA Block对应的是一个segment信息。同时需要注意的是，SSA区域主要是用于GC流程的，如果f2fs在mkfs的时候指定了只读属性，那么该区域也就没必要的。对应的是从SSA区域开始的所有segment的管理。SSA区域不做备份。

  • segment_count_main：Main Area区域对应的segment数，包括Node和Data的区域。

  • segment0_blkaddr：第一个segment的开始对应的block号，注意：本来第一个segment应该对应的是SuperBlock，但是f2fs是从CP区域开始对segment进行编码的，所以对应的是CP区域的开始block。

  • ⭐cp_blkaddr：CP区域的起始block号，和segment0_blkaddr一样。

  • ⭐sit_blkaddr：SIT区域的起始block号。

  • ⭐nat_blkaddr：NAT区域的起始block号。

  • ⭐ssa_blkaddr：SSA区域的起始block号。

  • ⭐main_blkaddr：Main Area开始的block号。

  • root_ino、meta_ino、node_ino：分别是root inode号，固定是3；元数据inode号，固定是2；Node对应的inode号，固定是1。其中root_ino是根目录的inode号，在文件系统挂载的是时候初始化，而meta_ino则对应元数据区域，缓存元数据的page cache；node_ino则对应的是所有node的page cache

  • uuid：随机值盐值，用于计算inode的crc。

  • volume_name：分区名。

  • extension_list：可以通过ioctl设定将哪些文件标记为cold，一般是读多写少的文件，例如：mp3、gif等等

  • version：mkfs时的linux的版本号。

  • feature：mkfs指定的特性列表，例如：加密、SuperBlock的crc检查等等。/sys/fs/f2fs/features/表示文件系统支持的特性。

  • encrypt_pw_salt：密钥的加密盐值，可通过ioctl获取。

  • devs：文件系统对应的设备列表，每个设备都有各自的地址访问，在写数据的时候需要遍历所有设备。

  • s_encoding、s_encoding_flags：文件名编码格式？

  • s_stop_reason：CP停止的原因（CP失败的原因？）

  • s_errors：文件系统损坏的原因。

2.3 SIT

SIT区域由N个struct f2fs_sit_block组成，每一个struct f2fs_sit_block包含了55个struct f2fs_sit_entry，每一个entry对应了一个segment的管理状态。每一个entry包含了三个变量: vblocks(记录这个segment有多少个block已经被使用了)，valid_map(记录这个segment里面的哪一些block是无效的)，mtime(表示修改时间)。

注意到内核代码里面有一个很有意思的注释：

在f2fs_sit_entry的第一个字段vblocks其代表的含义为：

  • **[15:10]**表示当前Segment类型是属于哪一种，HOT/WARM/COLD * NODE/DATA

  • **[9:0]**表示有效块的数量

  • 这2个值也对应了dump_sit中的值

首先我们验证一下上面的注释是否是正确的：

观察到第一个Segment 0的前2个字节为0xc01 = 0000 1100 0000 0001，其高[15:10]位为0000 11 = 3，低[9:0]位为00 0000 0001 = 1，可以看到下面的dump_sit中segno: 0对应的seg_type: 3，vblocks: 1

通过dump.f2fs -s 0~-1 f2fs.img 看一下SIT的整体情况

我们再来看一个有趣的现象，观察segno: 9，发现它是一个Warm Node，

这个segment偏移量为9，前面已经有9个Segment了，一个Segment为2MB，而Main Area的起始位置为0x100000，所以

  • SegNo 9的物理起始地址为：0x20000(2M) * 0x9 + 0x100000 = 0x1300000

又注意到e0这个位置前面总共59个字节，也就是472Bit，我们知道这是一个位图，其代表了前472个Block都是无效的，其大小总共为：

  • 472 * 4096 = 1,933,312Byte = 0x1D8000

下面开始看0xe0(11100000)其表示了从第472个Block开始的8个块，其中第472，473，474个Block是有效的，总共3个，这也验证了dump_sit的vblocks为3。

  • 计算出第472个Block的位置为0x1300000 + 0x1D8000 = 0x14D8000

2.4 NAT

这里一开始我很奇怪，为啥这里只有3个inode，即node_ino:1、meta_ino:2、root_ino:3，那其余文件的inode去哪里了，不是说NAT维护了一张inode no和block addr的表吗。

这里其实是由于Journal机制，为了不频繁读写NAT，所以将nat的改动先存放在了nat journal中，所以这也是为什么nat journal和NAT区域都是按照f2fs_nat_entry 排布的

只有当nat journal区域撑满了放不下了，才会刷写到磁盘对应的NAT区域中。

我们做个实验，写一个shell脚本f2fs.sh

功能就是创建2000个文件，文件名为hello[i]，向里面写入hello amx[i]

执行这个脚本，再次hexdump -C -s 0xA00000 -n 2097152 f2fs.img就可以看到NAT区域已经更新了【如果没更新，就把for循环次数放大】。后续直接 rm -rf hello*删掉这些文件即可
