主题：文件系统性能测试benchmark工具iozone

最近exfat的maintainer有一笔提交如下：https://lore.kernel.org/all/20260507124238.7313-1-linkinjeon@kernel.org/

我让AI对作者的测试结果进行了分析，并给出了测试命令

从你的结果（400MB/s 左右的吞吐）来看，这极大概率是使用了 O_DIRECT（绕过 Page Cache 的直接 I/O），因为如果是 Buffered I/O，1GB 的文件在 16GB 内存的机器上会被完全缓存，读取速度通常会飙升到 GB/s 级别。

以下是严格按照你的参数（1GB 文件，64KB 块大小）生成的复现命令：

  • 单线程测试命令 (1 Thread)

用于压测单文件、单线程的读写吞吐量：

  • -i 0: 执行 Write / Re-write 测试。
  • -i 1: 执行 Read / Re-read 测试。
  • -s 1g: 指定测试文件大小为 1GB。
  • -r 64k: 指定每次 I/O 请求的 Record Size 为 64KB。
  • -I: 开启 O_DIRECT 模式，绕过文件系统缓存，直接测试 VFS 到底层介质的路径。
  • -f: 指定测试文件的路径。
  • 多线程测试命令 (4 Threads)

为了准确模拟多线程并发行为，通常会让每个线程独立读写自己的文件，以观察文件系统级别的锁竞争（而不是单个文件的 inode 锁）：

  • -t 4: 启动 4 个并发线程。
  • -F ...: 为 4 个线程分别指定独立的测试文件。这确保了我们测试的是 exFAT 全局分配器和超级块操作的并发度。

注意： 如果你在跑测试时没有加 -I，那么你测出的可能是 Page Cache 刷脏页（Writeback）和预读（Readahead）的逻辑。如果确实是 Buffered I/O，去掉上述命令中的 -I 即可。

iozone工具介绍

iozone与fio比较

1.Fio (Flexible I/O Tester)：存储界的“瑞士军刀”

fio 由 Linux 内核块层（Block Layer）的维护者 Jens Axboe 编写。由于作者本身的背景，fio 天生就带着极其浓厚的“底层硬件”和“极限压测”基因。

  • 侧重点：极限性能、I/O 引擎、并发与延迟
    • 极度灵活的 I/O 引擎：支持 sync, libaio, posixaio, mmap, 以及现代极高性能的 io_uring。它甚至能绕过文件系统，直接对裸盘（如 /dev/nvme0n1）发起 I/O 请求。
    • 精准的控制力：它可以精确控制队列深度（iodepth）、读写比例（如 70% 读 30% 写）、块大小（block size）、甚至 I/O 的离散度（通过随机分布模型）。
    • 极客级别的数据统计：提供极其详尽的时延分布数据（如 P99, P99.99 尾部延迟），这对于评估现代 SSD 的垃圾回收（GC）机制至关重要。

2.IOzone：文件系统与缓存的“显微镜”

iozone 是一个历史悠久的老牌工具，它的核心是通过标准的 POSIX 系统调用（操作系统提供的标准文件操作接口）来全面评估文件系统（File System）的表现。

  • 侧重点：文件系统接口、Page Cache 行为、跨平台
    • 多维度的系统调用：专门测试操作系统的 read, write, mmap, pread, pwrite, fsync 等系统调用在不同情况下的表现。
    • 自动遍历的“测试矩阵”：iozone 最标志性的功能是它的全自动测试模式（-a 参数）。它会自动使用不同的“文件大小（File Size）”和“记录大小（Record Size / Chunk Size）”进行排列组合测试。
    • 洞察缓存机制：通过上述的“测试矩阵”，iozone 可以清晰地测出当文件大小从“能够完全放入内存缓存（Page Cache）”增长到“超过内存容量，必须直接读写磁盘”时，性能呈现的悬崖式下跌曲线。它生成的 Excel 数据非常适合绘制 3D 性能曲面图。

3.核心区别对比概览

维度 Fio IOzone
测试主要目标 裸块设备（SSD/HDD/NVMe）、I/O 调度器 文件系统（ext4, xfs, nfs）、OS 缓存（Page Cache）
I/O 发起方式 丰富且底层的引擎（libaio, io_uring 等） 标准的 POSIX API（read, write, mmap 等）
并发与队列 极强，支持设置极高的 iodepth 和多线程 相对较弱，主要关注单/多进程的 POSIX 并发
核心统计指标 IOPS、吞吐量、极微秒级的尾部延迟（Tail Latency） 不同“文件大小”与“记录大小”组合下的吞吐量
数据输出直观性 偏向于纯文本报表和直方图 可直接输出 Excel 格式，方便生成 3D 性能曲面图
