内核Notifier Chain 机制

内核Notifier Chain 机制

一句话理解

notifier chain 是内核里的“发布-订阅式事件广播机制”：某个子系统发生状态变化时，不需要硬编码调用所有关心者，而是沿着一条回调链逐个通知已注册的消费者。

1.notifier chain 解决了什么问题

内核中经常出现这样的问题：

  • 某个子系统发生了状态变化，例如网卡注册、内存热插拔、系统准备 suspend、设备进入 idle、系统 reboot。
  • 很多模块都可能关心这个事件，但事件源不应该知道所有消费者的细节。
  • 消费者可能来自内核核心代码，也可能来自模块；消费者数量也可能随配置变化。

notifier chain 的核心价值就是解耦。事件源只负责“广播事件”，消费者只负责“注册回调并处理事件”。事件源不需要硬编码调用 A、B、C 模块。

类比一句话

事件源像“广播站”，notifier head 像“广播频道”，notifier_block 像“订阅者登记表”，call_chain 像“开始广播”，callback 像“听到广播后的动作”。

2.核心数据结构与接口

核心结构是 struct notifier_block。它本质上是一个链表节点，里面放着回调函数、next 指针和优先级

字段 作用 理解方式
notifier_call 事件发生时被调用的回调函数 订阅者接到广播后的处理函数
next 指向下一个 notifier_block 订阅者链表
priority 优先级，数值越大越先调用 VIP 先处理，普通用户后处理

2.1 三类基础操作

操作 典型接口 含义
注册 atomic/blocking/raw/srcu_notifier_chain_register() 把 notifier_block 插入链表
注销 atomic/blocking/raw/srcu_notifier_chain_unregister() 把 notifier_block 从链表删除
广播 atomic/blocking/raw/srcu_notifier_call_chain() 沿链表逐个调用回调函数

3.四种notifier chain的区别

Linux 内核中 notifier chain 主要分为四类：atomic、blocking、raw、SRCU。它们的核心差别不是“功能”，而是“上下文约束”和“同步保护方式”。

类型 回调上下文 回调能否 sleep 内部同步 适合场景
Atomic atomic / interrupt 相关上下文 不能 注册/注销用 spinlock；call_chain 用 RCU 中断、原子路径、不能阻塞的快速通知
Blocking process context 可以 rw_semaphore 普通内核线程/进程上下文，回调可能需要睡眠
Raw 由调用者决定 由调用者保证 没有内置保护 调用者已经有自己的锁或上下文规则
SRCU process context 可以 注册/注销用 mutex；调用用 SRCU 调用非常频繁、注销很少发生的通知链

选择口诀

能不能 sleep 是第一判断条件：不能 sleep 选 atomic；能 sleep 通常选 blocking；调用特别频繁且注销很少，考虑 SRCU；你完全知道自己在干什么并且外部已有锁，才考虑 raw。

4.返回值语义

返回值 含义 是否影响后续回调
NOTIFY_DONE 我不关心或没有处理 不影响，继续
NOTIFY_OK 我处理了，结果正常 不影响，继续
NOTIFY_STOP 我处理了，并希望停止继续通知 停止
NOTIFY_BAD 我拒绝/否决这个动作，常用于可回滚事件 停止
notifier_from_errno(err) 把负 errno 转成 notifier 返回值 通常错误会带停止语义

关键点

NOTIFY_BAD 不只是“失败”，它常用于 veto 语义。例如 memory hotplug 的 GOING_ONLINE / GOING_OFFLINE 阶段，某个消费者返回 NOTIFY_BAD 可以阻止后续流程。已经完成且不可回滚的阶段，不能随意失败。
