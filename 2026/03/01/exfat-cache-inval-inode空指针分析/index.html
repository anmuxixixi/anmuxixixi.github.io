问题描述

最近遇到一个exfat的bug:

代码执行list_del_init的时候发现&cache->cache_list为空指针

slab分配器初始化问题

基础回顾

kmem_cache_create和kmem_cache_alloc这两个函数的关系就像是“模具生产线”与“生产具体零件”的关系。在 Linux 内核内存管理中，它们共同构成了 SLAB/SLUB 分配器的核心。

kmem_cache_create：创建“模具”（池子）

它的作用是向内核申请并定义一种新的对象类型的内存池。

  • 什么时候用：通常在驱动模块初始化（init）时调用一次。
  • 做了什么：它并不会立即分配大量内存，而是告诉内核：“我以后需要频繁申请这种固定大小（比如 sizeof(struct exfat_cache)）的对象，请帮我规划好管理方案。”
  • 关键参数：
    • 对象的大小。
    • 对齐方式。
    • **构造函数 (constructor)**：可选。定义对象在内存页首次划分为对象时如何初始化。

kmem_cache_alloc：生产“零件”（对象）

它的作用是从已经创建好的“模具”中取出一个具体的对象实例。

  • 什么时候用：在逻辑运行过程中，需要用到该结构体实例时调用（如：发现一个新簇，需要存入缓存）。
  • 做了什么：从 SLAB 池子里弹出一个空闲对象。如果池子空了，它会自动向伙伴系统（Page Allocator）申请新的物理页。
  • 关键参数：
    • 指向 kmem_cache_create 创建的那个 cache 指针。
    • 分配标志（如 GFP_KERNEL 或 GFP_NOFS）。
  1. 形象化对比表
特性 kmem_cache_create kmem_cache_alloc
比喻 建立一家 “螺丝加工厂” 从工厂里拿 “一颗螺丝”
执行频率 极低（驱动加载时运行一次） 极高（业务逻辑中反复调用）
内存消耗 仅占用极少量内核管理元数据 占用实际的对象内存空间
对应函数 kmem_cache_destroy (拆除工厂) kmem_cache_free (还回螺丝)
主要产出 一个 struct kmem_cache * 指针 一个指向具体对象的 void * 指针

问题分析

分析代码后发现，为slab内存分配问题

exfat_alloc_inode() 函数负责申请并初始化一个 Inode 对象。然而，在该函数中缺失了对缓存相关字段的初始化步骤。

由于 exfat_inode_cachep 使用了 Slab 分配器，当文件 A 被关闭释放后，其 Inode 内存块返回池中。当文件 B 被创建时，极大概率复用这块物理地址。

  • 后果：文件 B 的 ei->cache_lru 链表头依然保留着文件 A 销毁前的 next 和 prev 指针。

如果 Inode 在完全初始化（如开始读写并填充缓存）之前就被销毁（如设备强制卸载、分配失败导致的回滚），会触发清理逻辑：

  1. 判定逻辑：while (!list_empty(&ei->cache_lru))。
  2. 误判发生：由于 cache_lru.next 继承了旧指针（不等于自己），list_empty 返回 False。
  3. 空指针解引用：代码尝试通过 list_entry(ei->cache_lru.next, ...) 访问节点。
  4. Slab 的角色：Slab 分配器在释放对象时，通常会将内存头部（即旧指针指向的位置）抹掉或设置为 NULL（作为管理链表的一部分）。当代码尝试读取该地址时，立即触发 **Kernel Panic (NULL Pointer Dereference)**。

我们可以用“租用办公室”来理解这个过程：

  • 公司 A（旧 Inode）：搬走前在黑板（内存）上写下了“仓库后门在 A 街 100 号”。随后公司 A 把那个仓库退租了（Slab 释放了 Cache 节点）。
  • 保洁员（Slab 管理员）：公司 A 走后，保洁员把 A 街 100 号的招牌拆了，那里现在是一片空地（标记为 NULL）。
  • 公司 B（新 Inode）：搬进同一间办公室。因为 exfat_alloc_inode 没擦黑板，公司 B 以为自己真的在 A 街有仓库。
  • 崩溃：公司 B 打算去 A 街 100 号取货，结果发现那里是个大坑（空指针），直接掉进去了。

Q：为什么在exfat出现概率这么低？

A：正常在evict_inode的时候都已经对inode B进行了赋值。但是有一种情况就是inode b刚刚alloc，设备强制拔出后走到了evict_inode，也就是inode b还没来得及拔出呢，就去释放inode b的cache_list（而inode b是slab分配器分配的，其实指向的内存地址还是inode a的)

并且能说明一个问题：在slab分配内存时，

  • 验证了根本原因：exfat_alloc_inode 没有初始化 cache_lru 链表头，导致新分配的 Inode 带有旧内存的残留值。
  • 验证了触发机制：残留值（尤其是 0）会欺骗 list_empty() 宏，使其返回“非空”。
  • 验证了崩溃路径：随后的 list_for_each 或 list_del 动作会解引用这个残留值，造成不可避免的系统 Panic
