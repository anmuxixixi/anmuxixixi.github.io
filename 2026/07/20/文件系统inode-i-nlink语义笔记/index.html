文件系统inode->i_nlink语义笔记

Linux inode->i_nlink：何时为 0，何时不为 0

范围：Linux VFS 的 struct inode::i_nlink。本文按“语义类别”穷举状态，而不是枚举内核中每一种文件系统的每一个赋值点。具体文件系统可以按自己的磁盘格式设置 i_nlink，也可以在 getattr() 中另外设置返回给用户态的 st_nlink。

1. 先给出准确结论

i_nlink 是由具体文件系统维护、由 VFS 使用的链接计数字段。只看它是 0 还是非 0，不能无条件推出 inode 在当前命名空间中是否存在可见路径。

只有同时满足以下前提时，才能把它直接理解为目录项数量：

  1. inode 是已经完成创建并发布的、路径可命名的非目录对象；
  2. 文件系统采用传统 Unix 硬链接语义；
  3. 文件系统把该值作为实际链接数精确维护；
  4. 观察期间没有并发的 link、unlink 或 rename。

在这些前提下，非目录 inode 的 i_nlink == N 表示文件系统中有 N 个硬链接目录项关联到它。超出这些前提时：

  • i_nlink > 0 只表示当前链接计数字段是正数；它可能是实际链接数，也可能是 inode 初始化时设置的 1，或者是具体文件系统按自身规则设置的值；
  • i_nlink == 0 只表示当前链接计数字段为零；常见原因是最后一个硬链接已删除，或者 inode 原本就是匿名临时对象；
  • i_nlink == 0 不等于 inode 已释放。打开的文件描述符、内存映射、cwd 等引用仍可使 inode 和数据存活。

用户态的 stat.st_nlink / statx.stx_nlink 通常就是这个值。VFS 的 generic_fillattr() 将 inode->i_nlink 填入 kstat.nlink；文件系统也可以在自己的 getattr 中覆盖它。

这些例外具体指什么

这些情况应直接按照文件系统代码的实际处理来理解：

  1. ext4：启用 DIR_NLINK 后，如果目录的链接数无法继续由 16 位 i_links_count 精确表示，ext4 可以把该字段设为 1，用 1 表示“链接数未知”。这里的 1 不是实际链接数。
  2. FAT：FAT 磁盘格式没有 Unix i_nlink 字段。Linux FAT 驱动读取目录 inode 时调用 fat_subdirs(inode) 统计子目录，再把结果传给 set_nlink()；根目录使用 fat_subdirs(inode) + 2。因此这个 i_nlink 是 FAT 驱动按 VFS 需要设置的，不是从磁盘上的同名字段读取的。
  3. overlayfs：对 merged directory，overlayfs 可以在 getattr() 中执行 stat->nlink = 1。这改变的是本次返回给用户态的 st_nlink，不能据此断言 inode->i_nlink 也等于 1。

2. 内核中的基本规则

当前内核源码给出了四个关键事实：

  1. inode_init_always_gfp() 初始化新分配的内存 inode 时，先令 __i_nlink = 1。因此“刚分配、尚未发布到目录”的短暂内部状态也可能是 1；不能把 i_nlink > 0 无条件解释成用户已经能通过路径找到它。
  2. 文件系统只能直接读取 i_nlink，修改应使用 set_nlink()、clear_nlink()、inc_nlink()、drop_nlink()、inode_inc_link_count() 或 inode_dec_link_count()。
  3. drop_nlink() 做减一；clear_nlink() 直接置零。首次到达零时，VFS 同时更新 superblock 的待删除 inode 计数。
  4. 内核明确允许某些文件系统把链接数从 0 改回非 0；inc_nlink() 对 0 → 1 要求 inode 带有 I_LINKABLE。O_TMPFILE 后用 linkat(..., AT_EMPTY_PATH) 发布文件就是用户可见的典型路径。

因此，准确的判断是：

先把 i_nlink 当作文件系统维护的一个数值，再结合 inode 生命周期、对象类型和文件系统实现解释它。零值通常驱动“已解除链接、等待回收”的语义；正值不保证当前命名空间中存在可见或可达路径。

3. 非目录 inode：所有通用状态变化

这里的“非目录”包括常规文件、符号链接本身、FIFO、Unix 域 socket 的路径节点和设备节点。下表描述采用精确传统链接计数的本地文件系统上的正常 VFS 变化；初始化、匿名对象和异常路径已单独标明。是否允许创建或硬链接还受权限及具体文件系统限制。

操作或状态 目标 inode 的 i_nlink 说明
VFS 刚分配内存 inode 默认 1 内部初始化值；尚不保证目录项已经发布
成功创建并发布一个具名对象 通常 1 传统语义下有一个硬链接目录项
link(old, new) 成功 加 1 两个名字的 (st_dev, st_ino) 相同
删除多个硬链接中的一个 减 1，仍大于 0 该文件系统中仍有其他硬链接；不保证它们在当前命名空间中可见或可达
删除最后一个硬链接 变为 0 若仍打开，可继续 fstat/read/write
rename(old, new)，new 不存在 源 inode 不变 只是换名字；不是先 unlink 再 link 的可见中间态
rename(old, new) 覆盖已有文件 源 inode 不变；旧 new inode 减 1，可能为 0 打开的旧 new fd 仍指向被覆盖的 inode
RENAME_EXCHANGE 两边通常都不变 两个名字交换指向
O_TMPFILE 创建成功 0 从一开始没有目录项，但 fd 持有它
可链接的 O_TMPFILE 经 linkat(AT_EMPTY_PATH) 发布 0 → 1 O_EXCL 会禁止这种发布
memfd_create() 等匿名临时文件 通常 0 无目录名字；具体伪文件系统决定其 st_nlink
创建流程失败并回滚 最终清零并回收 可在文件系统错误路径或内核探针中看到，通常不是稳定用户态状态
从磁盘读入已删除/损坏的 inode 可能为 0 文件系统恢复、孤儿链表和一致性检查场景；不能当成正常可达文件

不会改变 i_nlink 的操作

  • open()、dup()、fork() 后继承 fd、通过 Unix socket 传递 fd；
  • mmap()、把文件作为可执行文件运行；
  • 创建符号链接：增加的是“符号链接 inode”自身的链接数，不会增加目标文件的链接数；
  • bind mount、不同 mount namespace 中出现路径别名；
  • reflink、普通复制：它们创建不同 inode，通常各自为 1；
  • 修改内容、权限、属主、xattr、时间戳。

例如，进程打开文件后执行 unlink()：

  1. 最后一个硬链接目录项被移除，文件系统把 i_nlink 降为 0；
  2. fd 仍通过 struct file -> path/dentry -> inode 保持引用；
  3. /proc/<pid>/fd/<n> 常显示 原路径 (deleted)，fstat(fd) 仍返回原 inode 且 st_nlink == 0；
  4. 最后一个相关引用释放后，文件系统的 evict_inode/删除路径截断数据并释放 inode。

4. 目录 inode

传统 Unix 目录不能由普通用户调用 link() 创建任意硬链接，否则会破坏目录树无环约束。目录链接数由 mkdir、rmdir、目录 rename 自动维护。

对精确维护传统计数的文件系统：

“2”可理解为目录自身的 . 和父目录中指向它的名字；每个直接子目录的 .. 再贡献一个链接。根目录的 . 和 .. 都指向根，结果也遵循 2 + 直接子目录数。

操作或状态 相关变化
新建空目录 新目录通常为 2；父目录加 1
在目录内创建普通文件/符号链接/FIFO 父目录不变
在目录内新建直接子目录 父目录加 1
删除空子目录 被删目录通常直接清为 0；父目录减 1
将目录改名到同一父目录且不覆盖 源目录和父目录均不变
跨父目录移动目录且不覆盖 源目录不变；旧父目录减 1，新父目录加 1
用目录覆盖同一父目录中的空目录 被覆盖目录变 0；父目录减 1
跨父目录移动并覆盖空目录 被覆盖目录变 0；旧父目录减 1；新父目录通常净变化 0
已打开的目录被 rmdir fd 上 fstat 可看到 0，直到引用释放

注意：上述算式是常见语义，不是所有文件系统对用户态的承诺。应用不应依赖目录 st_nlink - 2 来准确计算子目录数。

5. 零值与非零值的语义分类

对 Linux 中具有链接语义的 inode，零值的常见来源可以归纳为：

  1. 最后一个链接已删除：unlink、rmdir 或覆盖式 rename 使旧 inode 的链接计数降为零；它可能因为 fd、mmap、cwd、内核内部引用等继续存活。
  2. 天生匿名：O_TMPFILE、memfd 以及部分伪文件系统/内核内部匿名 inode 没有普通目录项。
  3. 尚未完成或正在回滚的内部状态：创建、链接、重命名的错误恢复路径可能先清零，再由 iput 回收。
  4. 文件系统恢复或异常状态：磁盘上的孤儿 inode、崩溃恢复中的待删除 inode、损坏元数据或网络文件系统的短暂不一致。

i_nlink != 0 的来源包括：

  1. 精确的普通文件计数：对正常、已发布的具名非目录对象，通常等于该文件系统记录的硬链接数；
  2. 精确的目录计数：采用传统规则的目录通常为 2 + 直接子目录数；
  3. 初始化值：尚未发布的 inode 也可能因 inode_init_always_gfp() 而暂时为 1；
  4. 文件系统自己的处理规则：ext4 大目录可用 1 表示“链接数未知”；FAT、网络文件系统和伪文件系统按各自实现设置 i_nlink；
  5. 实现细节中的隐藏链接：NFS “silly rename”等机制可能保留隐藏目录项，原路径消失后计数仍不为零。

因此不能从非零值单独推出以下任一结论：

  • 当前进程可以构造出访问该 inode 的路径；
  • 当前 mount namespace 中存在可见名称；
  • 数值等于用户态遍历目录后能数出的路径数；
  • inode 已经完成创建并对用户态发布。

6. 文件系统特例

ext4 的 DIR_NLINK

ext4 的磁盘字段 i_links_count 只有 16 位。启用 DIR_NLINK 后，拥有极多子目录的目录可以把该字段设成 1，表示“真实链接数未知”，而不是只有一个硬链接。因此：

overlayfs

overlayfs 必须把上下层内容作为一个视图提供给用户。对 merged directory，它可在 getattr() 中把用户态 stat.st_nlink 设为 1，因为逐个计算整个视图的子目录链接数不值得。这里改的是 stat->nlink，不能据此断言 overlay inode 内部的 i_nlink 也被设成 1。带 index 的非目录硬链接另有联合计数逻辑。所以在容器里看到 st_nlink == 1 的目录不一定异常。

FAT/VFAT

FAT 磁盘格式没有 Unix inode 和硬链接；Linux FAT 驱动在内存中构造 VFS 所需的 struct inode：

  • 普通文件沿用新 inode 的链接数 1，文件系统不提供 .link 操作，因此 link(2) 返回不支持；
  • 新目录显式设为 2，并维护父目录计数；重新读取目录时调用 fat_subdirs() 统计子目录并设置计数；
  • vfat_unlink() / vfat_rmdir() 删除成功后调用 clear_nlink(inode)，所以打开但已删除的 FAT 文件仍可在内存中看到 0；
  • FAT 的目录项删除标记不是一个持久的 Unix i_nlink=0 字段。零值主要是 Linux VFS 内存状态，不能在 FAT 磁盘目录项中寻找同名字段。

NFS 和其他网络文件系统

协议和缓存会使观察时序不同。经典 NFS 客户端可能把仍打开的文件改名为 .nfs...（silly rename），等关闭后再删除；这时“原名字已 unlink”不等于服务器 inode 已经没有名字。不要用单次 stat 推断远端回收时刻。

8. 常见误区

  1. “i_nlink == 0 就能 kfree(inode)”——错。 还要等所有引用消失，并走文件系统的回收流程。
  2. “fd 数量会反映到 i_nlink”——错。 多次打开、dup 都不增加硬链接数。
  3. “删掉符号链接会减少目标的 i_nlink”——错。 删除的是符号链接自己的 inode。
  4. “目录的链接数就是目录项总数”——错。 传统公式只关心直接子目录，不含普通文件；ext4 还可能用 1 表示“链接数未知”。
  5. “原路径消失就一定是 0”——错。 可能还有其他硬链接或 NFS 为仍打开文件保留的隐藏名字；overlayfs 还可能在 getattr() 中另行设置 st_nlink。
  6. “i_nlink > 0 就一定能在当前 namespace 找到路径”——错。 链接可能位于当前 namespace 不可达的位置；该值也可能处于 inode 初始化窗口，或由具体文件系统按自身规则设置。
  7. “磁盘字段永远等于内存 i_nlink”——错。 日志、延迟写回、恢复过程和网络缓存都会影响两者的观察时刻；FAT 磁盘格式则根本没有同名的 Unix 链接数字段。
