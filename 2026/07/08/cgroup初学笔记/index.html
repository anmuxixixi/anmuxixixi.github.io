cgroup初学笔记

cgroup初学笔记【Linux cgroup与Block IO子系统原理详解：从进程分组到block层限流】

本文面向内核存储方向学习，重点解释 cgroup 的核心原理，以及它如何与 Block IO 子系统配合，实现按进程组统计、限速、权重分配和延迟保护。
代码路径以 mainline Linux 6.x 思路为主，Android/common kernel 或 5.10 厂商内核可能有裁剪或实现差异。

1. cgroup 是什么

cgroup 全称是 control group，中文通常叫 控制组。

Linux 官方文档对 cgroup 的定义是：cgroup 是一种把进程按层级组织起来，并沿这个层级以可控、可配置方式分配系统资源的机制。cgroup 由两部分组成：cgroup core 和 controller；core 负责组织进程层级，controller 负责具体资源分配。

2. cgroup 的核心设计

cgroup 的设计可以抽象成三层：

画成图是这样：

flowchart TD
    A[task_struct<br/>进程/线程] --> B[css_set<br/>一组 subsystem state]
    B --> C1[cgroup_subsys_state<br/>cpu]
    B --> C2[cgroup_subsys_state<br/>memory]
    B --> C3[cgroup_subsys_state<br/>io / blkio]
    C1 --> D1[cpu controller 私有数据]
    C2 --> D2[memory controller 私有数据]
    C3 --> D3[blkcg<br/>Block IO cgroup 私有数据]

官方 cgroup v1 文档描述了这个基本实现：每个 task 有一个引用计数的 css_set 指针；css_set 里保存每个 cgroup subsystem 对应的 cgroup_subsys_state 指针；task 不直接指向每个 hierarchy 里的 cgroup，而是通过 css_set -> cgroup_subsys_state 间接找到。

2.1 为什么要有 css_set？

如果 task 直接保存每个 controller 的 cgroup 指针，那么每次访问 controller 状态都需要处理复杂的 hierarchy 关系。

内核选择了 css_set 作为中间层：

这样做的好处是：

  1. 访问快：性能关键路径可以快速拿到某个 controller 的状态。
  2. 共享状态：多个 task 如果属于完全相同的一组 cgroup，可以共享同一个 css_set。
  3. 迁移清晰：移动 task 到新 cgroup 时，本质上是切换 task 的 css_set。

在不同 cgroup 版本中名字不同：

cgroup v2 io controller io.stat, io.max, io.weight, io.latency, io.cost.qos, io.pressure

9.3 权重：io.weight

io.weight 用于表示 sibling cgroup 之间的相对 IO 权重。

例如：

含义不是“前台一定得到 10 倍带宽”，而是在设备竞争时，前台拥有更高权重。

权重控制通常需要具体策略配合，例如 BFQ 或 iocost。

9.4 延迟保护：io.latency

io.latency 是 cgroup v2 的 IO workload protection 机制。配置一个目标延迟后，如果受保护 group 的平均延迟超过目标，controller 会 throttle 低优先级 peer group。官方文档对 io.latency 的描述正是：给 group 设置 latency target，如果平均延迟超过 target，就限制延迟目标更低优先级的 peer。

抽象逻辑：

图示：

flowchart TD
    F[foreground<br/>target=20ms] --> L{avg_lat > target?}
    B[background] --> Q[request_queue]
    F --> Q

    L -- 否 --> N[不干预]
    L -- 是 --> T[throttle background]
    T --> R[foreground 获得更低延迟]
