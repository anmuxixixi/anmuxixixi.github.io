IOPS和带宽

  • 每秒完成大约2076次4K随机写IO
  • 随机写带宽大约8310KiB/s，也就是8.1 MiB/s

这个带宽和IOPS是对应的：2076 IOPS * 4 KiB ≈ 8304 KiB/s

  • 1461MiB/180061msec：表示 180 秒内总共完成了大约 1461 MiB 写入。

注意：虽然每个测试文件是 2GiB，4 个 job 总测试空间是 8GiB，但因为你用了 time_based=1，fio 是在 8GiB 范围内随机覆盖写，跑满 180 秒就结束，所以最终实际完成写入量是 1461MiB，而不是必须写满 8GiB。

延迟指标：slat/clat/lat

  • slat（submission latency）：fio 从用户态提交 IO 到内核队列所花的时间【平均耗时68.25 us】
  • clat（completion latency）：IO 从提交到真正完成的时间，主要反映磁盘/块层/云盘后端的耗时【平均耗时61.5ms】
  • lat = slat + clat：为两者的总延迟

p99/p99.9指标

这部分最重要：

  • 50.00th=[ 61]：一半IO延迟低于61ms，也就是 p50
  • 95.00th=[ 69]：95%的IO延迟低于69ms
  • 99.00th=[ 142]：99%的IO延迟低于142ms
  • …
  • 99.90th=[ 207]：99.9%的IO延迟低于207ms

极端尾延迟已经接近 271ms，对于 4K 随机写来说，这个尾延迟偏高

latency bucket

表示不同延迟区间内的 IO 占比。

  • 关键是：lat (msec): 100=97.27%，说明绝大多数 IO 落在 50ms 到 100ms 这个范围附近。
  • lat (msec): 250=1.59%：说明还有一部分 IO 延迟落到 100ms 到 250ms 这个范围。（这和前面的 p99=142ms、p99.9=207ms 是匹配的。）

IO Depth

说明fio的每个job 基本都维持在 iodepth=32。

你有4个job，所以总outstanding IO 大概是：4 * 32 = 128

磁盘状态Disk stats

  • util=99.95%：设备繁忙率；说明这块盘几乎100% 忙。fio已经把盘打满了
  • ios=0/426304：读IO是0，写IO是426304【vda上块层统计到了426304 次写 IO，没有读 IO】
    • issued rwts: total=0,373956,0,0 short=0,0,0,0 dropped=0,0,0,0
    • ios=0/426304, merge=0/72548, ticks=0/25745767, in_queue=25746025, util=99.95%
    • fio issued是fio线程发出的逻辑IO数量（373956）
    • Disk stats是块设备层最终看到的设备IO数量（426304）
    • 两者不一样是正常的，中间可能多出：文件系统元数据写入、journal写入、系统盘上其他进程的后台IO、块层拆分或重新组织后的IO
  • merge=0/72548：有一部分写请求在块层发生了合并（读合并次数 = 0；写合并次数 = 72548）
  • in_queue=25746025(ms)：总的队列等待/服务时间很高，也印证了磁盘处于饱和状态。
    • in_queue对应Linux块设备统计里的weighted time spent doing I/Os
    • 它统计的是：某一时刻有多少个IO正在队列中等待或被服务 乘以 这段时间持续了多久
    • in_queue最有用的是用它反推平均队列深度【平均队列深度 ≈ in_queue / fio运行时间】
    • 代入数据：25746025 / 180061 ≈ 143,也就是说，这次fio测试期间，vda块设备层平均大约有143个IO处于等待；这和fio参数outstanding IO = 128基本吻合；
    • 设备层看到143，比128稍高，可能来自：文件系统元数据写入、系统盘其他后台 IO、请求合并/拆分
